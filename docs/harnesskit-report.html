Context Harness MVP

把现有仓库升级为
Agent 可稳定协作的研发环境。

HarnessKit 面向正在引入 AI Agent 的研发团队，帮助他们把项目背景、仓库结构、工程规则和验证入口沉淀为可版本化的 Context Harness。 通过 CLI 注入 AGENTS、RULES、Skills 与 Validations，并配合静态分析和防漂移检查， 团队可以用更低成本完成 Agent 接入，让每一次协作都从清晰上下文、明确边界和确定性反馈开始。

Harness 核心资产目录
AGENTS.md 顶层指南路由器，引导 Agent 进场并明确收敛步骤。
ARCHITECTURE.md 粗粒度仓库地图，帮助 Agent 快速定位核心代码与边界职责。
RULES.md 规则事实索引，记录跨开发任务、不可被破坏的短规则约束。
.agents/skills/ 按需加载的任务指令集（如 scan-stack, fill-rules 等），承载特定流程。
Validations 定义能被自动化执行的各类 Probes，反馈执行状态并绑定 Runner。

Problem Statement

两大痛点：不懂如何编写，
以及无力进行长期维护。

第一，传统团队没有为 Coding Agent 编写结构化规则与运行地图的经验与直觉； 第二，即使人肉编写了一套上下文说明，随着项目代码的日常迭代，上下文资产也会因人工维护的巨大成本而迅速发生漂移、分叉与腐烂。

Definition / 什么是 Harness
在一切皆漂移的概率开发系统中，Harness（安全带/脚手架）的本质不是具体的开发路径（Workflow），而是约束、护栏与纠偏机制。 HarnessKit 并不去控制 Agent 的思维过程，而是通过 Git + Hook 这道唯一的门，控制它最终向仓库提交了什么，从而以确定性应对概率模型的漂移。 “Scan 负责测绘地形（回答物理世界是什么），Harness 负责安装约束（提供不偏离轨道的护栏），而 Workflow 负责画出路径。”
1 个面向 Agent 的主入口路由器 (AGENTS.md)，实现全景路由。
17 条已启用的本仓库 Rule Details 规则文件，确保约束有据可查。
50 项 Pytest 自动化测试，实现 CLI 初始化、Linter 机制的 100% 覆盖。
1 个一致的本地完整验证入口：执行 make verify 触发自检。

Shipped Content

从一纸设计，
到真正可落地部署的 Harness 引擎。

这一阶段重点对齐了 CLI 功能、本地 Skills、Rule Linkage 检查，并且通过 make verify 达成了完整的验证 Receipt 机制。

Template Output

默认安装 Context Harness

生成并写入包括 AGENTS.md、ARCHITECTURE.md、RULES.md、.harnesskit/facts.md 事实清单、15 条基础规则 details 以及 Codex 技能描述。

Local Skills

明确分工的 Codex 本地技能集

在 .agents/skills/ 下提供 scan-facts（扫描事实）、scan-stack（检测技术栈）、fill-*（将事实写回各入口）以及代码修改验证技能，规范 Agent 操作路径。

Validation Runner

make verify 写入 Receipt 凭据

校验运行结果（含时间戳和详细输出）会自动序列化成 JSON 写入 .harnesskit/receipts/runs/，作为 Agent 和 Reviewer 评估仓库健康的证据。

✓AGENTS/RULES 结构完全对齐：本仓库实现了严谨自举，核心规则均对应 detail 且格式严密。
✓规则细节一致性：RULES.md 中的每条规则都严格与 .harnesskit/rules/RULE-*.md 对应，并包含了 Runner 绑定细节。
✓文件与脚本收敛：清理了过渡期文件，所有的验证统一收敛至 run_validation.py 进行分发。
✓Linter 强化：Harness-linter-poc 增加了对规则细节引用链（Linkage）以及技术栈事实同步状态的严密检查。

Operating Loop

从一次初始化，
到 Agent 工作闭环的自我循环。

HarnessKit提倡的工作流核心是：通过频繁、小幅的本地扫描与填充，使 Agent 能够持续地从仓库提取真实事实，自我校验并纠正，最终在开发结束前达成自检通过。

1

Init (配置初始化)

使用 harnesskit init 一键安装 Context Harness 核心资产模板与 Codex 本地技能。

2

Scan (事实扫描)

Agent 在遇到开发任务时，根据 skill 描述扫描依赖与命令事实，并更新 facts.md。

3

Fill (入口沉淀)

调用 fill-* 技能，将最新事实与规则变动写回 AGENTS.md、RULES.md，形成稳定入口。

4

Validate (命令验证)

运行 make verify 自动触发完整自检，生成具有时间戳的验证 Receipt 确认仓库无隐患。

5

Preserve (质量防腐)

由 pre-commit 钩子自动触发 Harness Linter POC，拦截所有断链、过期规则或定义漂移。

Future Roadmap

专注做好 Context Preservation，
不盲目扩张产品范畴。

我们将继续坚定地把当前 MVP 的核心边界控制住。不研发重度的编排和运行时沙箱，确保 HarnessKit 作为一个轻量级、自解释、易上手的工具包存在。

近期：Harness Preservation 质量稳固

让 Harness Linter POC 具有更丰富的错误自动修复（autofix）能力，使规则与 Linkage 校验真正做到无感阻断与轻快修复。

中期：丰富 Validation Probes

从实际项目事实出发，沉淀更多适用于通用技术栈的规则与校验 Probes，使验证能力不仅有，而且深。

扩展：适配更多集成（Integrations）

待 Codex 集成充分打磨并定型后，考虑接入 Claude Code 以及企业内部的其他智能编码环境集成。

非目标 (Out of Scope)

不开发 Agent Loop 编排层，不管理虚拟沙箱与多 agent 调度，避免产品焦点在开发早期偏离与失控。

Decision Points

本阶段汇报期待达成的共识：

我们希望在以下三个关键维度与团队达成高度一致，从而心无旁骛地在后续阶段加速推进 Context Harness 的体验沉淀。

1. 紧扣 Context Harness 边界

坚守 MVP 范围，优先帮传统团队解决上下文“冷启动编写”与“演进防腐烂”，不做泛化的 CI/CD 平台替代品。

2. 下阶段演进焦点

以 Linter 的稳定度与自动生成/自动修复（autofix）作为核心衡量标准，形成更紧密闭环。
