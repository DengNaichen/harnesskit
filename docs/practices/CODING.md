# Coding Practices

本文件记录 HarnessKit 的代码判断指导。它不是硬规则索引；必须遵守的约束仍以 [RULES.md](../../RULES.md) 和 `.harnesskit/rules/` 为准。

## 判断原则

- 优先遵循现有模块边界、命名习惯和测试风格；小改动不要引入新架构。
- 先让现有系统教你怎么改，再决定是否需要抽象。
- 抽象应减少真实重复、降低认知负担或表达稳定边界；不要为了“看起来完整”提前抽象。
- 用户可见行为、模板行为和 lint 行为的测试应验证结果，而不是锁死内部实现。
- 代码评论只解释不明显的边界、风险和原因；避免重复代码本身已经表达的内容。

## Review Questions

- 这个改动是否保持在正确的 runtime、template、linter 或 docs 边界里？
- 新 helper 是否真的降低复杂度，还是只是把逻辑搬到别处？
- 测试是否覆盖了用户能观察到的行为变化？
- 是否有无关重排、格式 churn 或顺手重构混入？
- 失败路径、错误消息和边界条件是否与现有风格一致？

## 和 Rules 的关系

违反现有模块边界、测试约束或验证入口时，按对应 Rule 处理；本文件只帮助判断“怎样写得更像这个仓库”。
