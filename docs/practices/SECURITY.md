# Security Practices

本文件记录 HarnessKit 的安全判断指导。安全硬约束仍应抽到 [RULES.md](../../RULES.md) 和 `.harnesskit/rules/`，并尽可能绑定测试、lint 或 review。

## Scope

HarnessKit 当前主要风险来自 CLI 文件写入、模板生成、报告/context 资产、依赖和 agent 对仓库事实的处理。它不是生产服务，也没有当前已确认的安全披露流程或支持版本政策。

## Guidance

- 不要把 secret、token、私有凭据、真实用户数据或机器特定敏感信息写入模板、facts、rules、报告或生成产物。
- 写入目标仓库文件时，默认不覆盖已有文件；只有显式 `--force` 或等价确认才覆盖。
- 任何目标路径、模板路径或生成路径都应保持在预期项目目录内，避免路径逃逸。
- 报告和诊断输出应优先引用相对路径，避免不必要地暴露本机绝对路径。
- 新增依赖前确认用途、维护成本和锁文件同步，不把依赖存在误写成安全或发布保证。
- 安全政策、披露渠道、支持版本和响应 SLA 未配置时，应写成未配置或待确认，不要凭经验补全。

## Review Questions

- 输出里是否可能包含真实 secret、token、私有 URL、用户名或机器路径？
- 文件写入是否可能覆盖用户手写内容或写到项目外？
- 生成模板是否会诱导目标仓库提交敏感信息？
- 依赖、脚本或外部命令是否新增了未说明的信任边界？
- 安全声明是否有仓库证据，还是只是通用最佳实践？
