# ARCHITECTURE.md 设计说明

`ARCHITECTURE.md` 是给 agent 和贡献者使用的仓库地图。它不应该成为完整设计文档、API 参考、文件清单或历史说明，而应该帮助读者在修改前快速判断：某个职责属于哪里、哪些路径是关键入口、哪些边界不应该被随手跨越。

本文件只说明 `ARCHITECTURE.md` 应该如何设计，不记录当前仓库自己的完整架构地图。具体项目的架构地图应写在该项目根目录的 `ARCHITECTURE.md` 或同类文件中。

## 核心定位

`ARCHITECTURE.md` 的核心定位是 agent-facing context 的仓库地图。它回答“代码和文档在哪里、各自负责什么、修改某类行为应该先看哪里”。

一个好的 `ARCHITECTURE.md` 应能快速回答：

- 主要源码、测试、模板、文档和配置目录在哪里。
- 每个关键目录或文件负责什么。
- 哪些路径属于运行时代码、生成输出、测试、工具链、agent harness 或设计文档。
- 修改某类行为时，agent 应该先读哪些入口。
- 新增、删除或移动重要模块时，哪些地图条目需要同步。
- 哪些边界不应该混淆，例如运行时代码和 POC、模板输出和项目自身文档、设计说明和当前实现状态。

它不负责说明 agent 应该按什么流程工作，那是 `AGENTS.md` 的职责；也不负责定义工程规则和 Guard，那是 `RULES.md` 的职责；更不负责承载长篇产品背景和设计讨论，那些应放在 `README.md` 或 `docs/` 中。

## 设计原则

### 粗粒度优先

架构地图应记录会影响 agent 定位的路径，而不是枚举每个 helper 函数。目录、模块、重要入口文件和生成资产边界值得记录；纯内部实现细节只有在会改变修改入口或职责边界时才需要加入。

### 以职责写路径

每个路径说明都应回答“它负责什么”或“什么时候该看它”。不要只写“包含相关文件”这种空泛描述。更好的写法是说明这个目录和行为边界的关系，例如模板目录会影响目标仓库生成输出，测试目录覆盖初始化和 integration 安装行为。

### 不复制其他文档

`ARCHITECTURE.md` 可以链接到 `README.md`、`AGENTS.md`、`RULES.md`、skills 和设计文档，但不应该吞掉它们的内容。它只维护地图和职责边界：

- 产品定位放在 `README.md`。
- agent 工作流程放在 `AGENTS.md`。
- 工程规则和 Guard 放在 `RULES.md`。
- 具体技能流程放在 `.agents/skills/*/SKILL.md`。
- 长期设计讨论放在 `docs/`。

### 以仓库事实为准

架构地图必须来自真实路径、源码、测试、配置、脚本和文档。不要从模板、示例项目或预期规划中虚构目录职责。暂时无法确认的职责应保留待确认，而不是写成事实。

### 显式标出边界

当仓库里有容易混淆的区域时，架构地图要主动切清边界。例如：

- 打包运行时代码和实验性 POC。
- 当前项目文档和会安装到目标仓库的模板。
- 设计说明和当前项目状态。
- 本地验证 runner 和 CI/platform gate。

这些边界能减少 agent 把改动放错位置的概率。

## 覆盖检查

如果项目使用 harness linter 或类似工具，可以用 marker 表示某个路径的直接子项也应出现在架构地图中：

```md
- [`src/example/`](src/example/) <!-- harnesskit:coverage=direct-children -->：运行时代码入口。
```

这类覆盖检查适合用于少数关键目录，例如运行时代码、linter rule 分组或重要测试分组。不要给所有目录都加覆盖要求，否则地图会变成机械文件清单。

覆盖检查只能证明路径存在和子项被列出，不能证明职责描述准确。职责准确性仍需要 review 或 agent 判断。

## 推荐结构

一个实用的 `ARCHITECTURE.md` 通常包含：

1. 文件定位：说明这是粗粒度仓库地图，不是工作流规范或 API 文档。
2. 顶层地图：列出主要源码、测试、模板、文档、工具和 agent harness 目录。
3. 关键文件：列出包清单、agent 指南、规则文件、README 等高频入口。
4. 生成资产：如果项目会生成或安装文件，说明这些输出来自哪里、写到哪里。
5. 当前包形状：简短说明运行时代码目前的模块边界。
6. 相关文档：链接到 README、AGENTS、RULES、设计文档和路线图。

结构可以随项目调整，但应保持“地图优先”。如果某一节开始解释大量流程、规则或背景，通常说明内容应该转移到其他文档。

## 模板和项目版的差异

模板版 `ARCHITECTURE.md` 不能假设目标仓库已经有某些目录、语言或工具。它可以提供结构骨架、coverage marker 示例和 `[NEEDS CLARIFICATION: ...]` 占位符，但项目事实必须等待目标仓库补全。

项目版 `ARCHITECTURE.md` 则应该更具体：

- 写真实路径和真实职责。
- 删除已补全的占位符。
- 只保留对 agent 定位有价值的入口。
- 重要路径移动后同步更新链接。
- 对无法确认的职责保留短 TODO，而不是写成事实。

## 更新时机

以下变化通常需要同步检查 `ARCHITECTURE.md`：

- 新增、删除或重命名主要目录、模块或关键入口文件。
- 运行时代码、模板、测试、文档或工具链职责发生变化。
- 新增重要生成资产或改变生成目标路径。
- 引入新的 POC、integration、skill 或验证 runner。
- `AGENTS.md`、`RULES.md` 或 skills 中引用了新的重要路径。
- linter coverage hint 失败，说明地图和仓库结构已经漂移。

如果只是移动内部 helper，且不改变 agent 首先应该查看的位置，可以不更新架构地图。

## 自举项目的边界

当项目本身正在设计 context harness 时，设计文档和当前地图要分开：

- 设计文档说明 `ARCHITECTURE.md` 应该怎么设计。
- 根目录 `ARCHITECTURE.md` 说明当前仓库真实的目录职责。
- 模板 `ARCHITECTURE.md` 如果存在，应说明目标仓库初始化后如何补全自己的架构地图。

这三层不要互相复制。设计文档保持抽象稳定，项目地图跟随仓库事实更新，模板则保持保守和待确认。
